# AEGIS: Zero-Trust Attendance System
**Comprehensive Engineering & Security Project Report**

---

## 1. INTRODUCTION

### 1.1 Problem Statement
In large academic and corporate institutions, tracking attendance accurately is a logistical nightmare. Traditional methods like roll calls are time-consuming, while modern alternatives (like swiping ID cards or basic Bluetooth-based check-ins) are inherently flawed and highly susceptible to proxy attendance (buddy punching). Students or employees can easily share ID cards, spoof MAC addresses, or pass credentials to colleagues to fake their presence.

### 1.2 The Failure of Traditional Systems
*   **RFID/ID Cards**: Can be handed to a friend.
*   **Basic Biometrics**: Simple face recognition APIs are easily bypassed using static photographs or digital screens.
*   **Simple GPS Check-ins**: Easily defeated using "Fake GPS" applications or developer-mode mock locations.

### 1.3 Objective of AEGIS
The Automated Entry Geo-fenced Identification System (AEGIS) was engineered to solve these vulnerabilities by adopting a strict **Zero-Trust Architecture**. The system operates on the principle that *proximity is not presence*. To successfully mark attendance, AEGIS requires simultaneous cryptographic, spatial (GPS), biometric (Face), and temporal proofs, rendering proxy attendance virtually impossible.

---

## 2. SYSTEM OVERVIEW

AEGIS is a distributed, multi-tier application designed to securely capture and validate attendance in real-time. It eliminates human intervention by automating identity and location checks through a secure pipeline.

### High-Level Architecture Flow
```text
[Mobile App] → Captures GPS, Face Burst, Device ID, QR Data
      ↓
[Mobile App] → Signs payload with HMAC-SHA256
      ↓
[API Backend] → Validates Signature, Nonce, Timestamp, Rate Limits
      ↓
[AI Service] → Computes Liveness & ArcFace Embeddings
      ↓
[Database] → Verifies Cosine Similarity (Supabase) & Persists Record (MongoDB)
```

## 3. ARCHITECTURE DESIGN

AEGIS is composed of four distinct, decoupled components:

### 3.1 Mobile App (React Native)
The client-side application acts as the untrusted data collector. Its responsibilities include:
*   **QR Scanning**: Decodes dynamic session payloads generated by the teacher's dashboard.
*   **Data Collection**: Captures high-accuracy GPS coordinates, hardware Device ID, and a rapid burst of face images.
*   **Cryptographic Assembly**: Bundles all collected data, generates a unique nonce, appends a timestamp, and signs the entire request to prevent tampering in transit.

### 3.2 Backend (Node.js / Express)
The Node.js backend serves as the core orchestrator and the first line of defense (the DMZ).
*   **Request Validation**: Blocks malformed JSON and enforces strict schema validation.
*   **HMAC Verification**: Re-calculates the signature to ensure payload integrity.
*   **Rate Limiting**: Protects against brute-force and Denial of Service (DoS) attacks.
*   **Anomaly Detection**: Checks for impossible travel velocities and device-multiplexing.

### 3.3 AI Service (Python / FastAPI)
A specialized microservice strictly handling heavy compute tasks for biometric validation.
*   **Face Embedding Generation**: Extracts 512-dimensional vectors using the ArcFace model.
*   **Liveness Detection**: Analyzes image textures and temporal changes to reject photos and screens.
*   **Similarity Matching**: Compares the live embedding against the trusted baseline vector.

### 3.4 Databases
AEGIS employs a polyglot persistence strategy, utilizing the right database for the right task:
*   **MongoDB**: Document store for high-throughput, unstructured operations (Users, Sessions, Attendance Logs).
*   **Supabase (PostgreSQL + pgvector)**: Specialized vector database used exclusively for rapid, mathematically complex Nearest-Neighbor searches on biometric embeddings.

## 4. DATABASE DESIGN

### 4.1 MongoDB Collections
MongoDB handles the operational state of the application. Key collections include:
*   **Users**: Stores credentials, RBAC roles, and hashed device bindings.
*   **Sessions**: Stores active lecture metadata (geolocation, radius, validity window).
*   **Attendance**: The final, immutable ledger of successful check-ins.

*Schema Example (Attendance Collection):*
```javascript
const attendanceSchema = new mongoose.Schema({
  email: { type: String, required: true, index: true },
  sessionId: { type: mongoose.Schema.Types.ObjectId, required: true },
  timestamp: { type: Number, required: true },
  deviceId: { type: String, required: true },
  status: { type: String, enum: ['PRESENT', 'LATE'], default: 'PRESENT' }
});
// Compound unique index enforces strict DB-level idempotency
attendanceSchema.index({ email: 1, sessionId: 1 }, { unique: true });
```

### 4.2 Supabase (PostgreSQL + pgvector)
PostgreSQL, enhanced with the `pgvector` extension, is used to store and query facial embeddings. Relational databases inherently struggle with high-dimensional arrays, but `pgvector` allows us to perform native Cosine Distance calculations directly in SQL, avoiding the need to load thousands of vectors into memory.

*SQL Schema Example:*
```sql
CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE face_embeddings (
  user_id UUID PRIMARY KEY,
  email TEXT UNIQUE NOT NULL,
  embedding vector(512),
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- Enable Row Level Security (RLS)
ALTER TABLE face_embeddings ENABLE ROW LEVEL SECURITY;
```

## 5. CORE SECURITY FEATURES

### 5.1 Zero-Trust Model
AEGIS assumes the mobile device is compromised. No data (GPS, time, identity) is trusted at face value until it passes cryptographic and behavioral validation on the server.

### 5.2 Full Payload Signing
To prevent Man-in-the-Middle (MitM) attacks and data tampering, the entire request body is stringified and signed using HMAC-SHA256.

*Code Snippet:*
```javascript
// Mobile Client Payload Generation
const payload = JSON.stringify({
  email: user.email,
  sessionId: qrData.sessionId,
  timestamp: Date.now(),
  nonce: generateUUID(),
  gps: { lat, lon }
});

const signature = crypto.createHmac('sha256', APP_SECRET)
                        .update(payload)
                        .digest('hex');
```

### 5.3 Replay Protection
A malicious actor could intercept a valid request and resend it later. AEGIS prevents this by:
1.  **Timestamp Drift**: Rejecting requests older than 30 seconds.
2.  **Nonce Tracking**: Storing the unique `nonce` in a TTL (Time-To-Live) database. If the same nonce is seen twice, the request is dropped.

### 5.4 Device Binding
Upon registration, the student's hardware `deviceId` is cryptographically hashed and bound to their account. If Student A attempts to log in on Student B's phone, the backend detects the hardware mismatch and flags the anomaly.

### 5.5 Rate Limiting
Strict limits are applied per endpoint and per IP using `express-rate-limit`, ensuring AI compute resources cannot be exhausted by malicious scripts.

### 5.6 Behavioral Detection
AEGIS calculates "geo-velocity" by checking the distance and time between a user's consecutive check-ins. If a user marks attendance in Building A, and 5 seconds later attempts to mark attendance in Building B (3 miles away), the system detects impossible travel velocity and rejects the request.

## 6. FACE RECOGNITION SYSTEM

### 6.1 ArcFace Embeddings
AEGIS utilizes ArcFace (via the DeepFace library), a state-of-the-art deep learning model that maps facial features into a 512-dimensional hypersphere. This approach is highly resilient to changes in lighting, facial hair, and minor aging.

### 6.2 Cosine Similarity
To compare two faces, the system calculates the angle between their 512-D vectors. A smaller angle implies higher similarity.

### 6.3 Threshold Logic
A strict distance threshold (e.g., `< 0.22`) is enforced. Anything above this threshold is considered a mismatch.

### 6.4 Liveness Detection (IMPORTANT)
To defeat printed photos and digital screens, AEGIS captures a 3-image burst over 300ms. The liveness engine evaluates:
*   **Moiré Detection**: Uses Fast Fourier Transform (FFT) to detect the pixel grid patterns emitted by digital screens.
*   **Motion Consistency**: Tracks micro-movements across the burst frames.
*   **Embedding Consistency**: Ensures all frames belong to the same person.

*Composite Liveness Scoring:*
```text
Liveness Score = (0.45 * moire_confidence) + (0.30 * motion_variance) + (0.25 * embedding_consistency)

If Liveness Score < LIVENESS_THRESHOLD:
    Reject as Spoof Attempt
```

## 7. REQUEST FLOW (STEP-BY-STEP)

The lifecycle of a secure check-in is strictly ordered:
1.  **QR Scan**: App reads dynamic session ID and location constraints.
2.  **Data Capture**: App locks GPS coordinates and triggers the camera for a 3-frame burst.
3.  **Payload Signing**: App packages data, appends timestamp/nonce, and generates HMAC signature.
4.  **Backend Validation**: Node.js verifies signature, timestamp drift, nonce uniqueness, and device binding.
5.  **AI Verification**: Images are proxied to Python FastAPI. Liveness is calculated. If live, embedding is generated and matched against Supabase.
6.  **DB Write**: Upon AI success, Node.js performs an idempotent write to MongoDB, finalizing attendance.

## 8. ERROR HANDLING & RESILIENCE

Distributed systems are prone to network failures. AEGIS implements strict timeout enforcement to prevent the Node.js event loop from hanging if the Python AI service slows down.

*Code Snippet (Promise Race for Timeout):*
```javascript
const verifyFaceWithTimeout = async (imageBuffer) => {
    const aiCall = axios.post('http://ai-service:8000/verify', imageBuffer);
    const timeout = new Promise((_, reject) => 
        setTimeout(() => reject(new Error("AI Service Timeout")), 5000)
    );

    try {
        // First to resolve/reject wins
        return await Promise.race([aiCall, timeout]);
    } catch (error) {
        logSystemFailure(error);
        throw new HttpException(503, "Biometric validation temporarily unavailable");
    }
};
```

## 9. SECURITY AUDIT SUMMARY

Prior to deployment, AEGIS underwent rigorous internal auditing:
*   **Replay Attack Testing**: Intercepted payloads were resubmitted. 100% blocked by TTL Nonce validation.
*   **Spoofing Tests**: High-res iPad photos were presented to the scanner. 100% rejected by FFT Moiré analysis.
*   **API Abuse Testing**: Apache Bench was used to flood `/api/attendance` with 1,000 concurrent requests. Rate limiters successfully throttled malicious IPs, while MongoDB's unique compound indexes prevented any duplicate entries under heavy race conditions.

---

## 10. RE-AUDIT & IMPROVEMENTS

Based on initial audit findings, several architectural upgrades were implemented:
*   **Full Payload Signing**: Upgraded from signing just the session ID to signing the entire JSON body, eliminating parameter tampering.
*   **Key Rotation Protocols**: Abstracted `APP_SECRET` to environment variables, allowing seamless cryptographic key rotation.
*   **Timeout Enforcement**: Implemented hard 5-second limits on inter-service communications to prevent cascading failures.

---

## 11. LIMITATIONS

A transparent engineering assessment must acknowledge current limitations:
*   **No Hardware Attestation**: Without integrating Google Play Integrity or iOS App Attest, highly sophisticated rooted devices could theoretically patch the React Native runtime.
*   **Limited Monitoring**: Lacks a centralized observability stack (like ELK or Datadog) for deep log querying.
*   **No Distributed Queues**: Relies on synchronous HTTP calls between Node.js and Python, which limits extreme burst scalability compared to an asynchronous Kafka/RabbitMQ pipeline.

---

## 12. FUTURE ENHANCEMENTS

To scale AEGIS to an enterprise level, the following roadmap is proposed:
*   **Play Integrity API**: Cryptographically verify the mobile device hardware is not rooted or running an emulator.
*   **Redis Rate Limiting**: Move rate limiting and Nonce tracking from memory/MongoDB to Redis for microsecond latency.
*   **Alerting System**: Integrate Prometheus and Grafana to trigger PagerDuty alerts upon anomalous spikes in failed biometric validations.

---

## 13. CONCLUSION

AEGIS demonstrates that academic attendance tracking can be elevated to enterprise security standards. By combining cryptographic payload signing, geospatial boundaries, and advanced AI biometric validation, the system successfully nullifies proxy attendance. 

The architecture balances the heavy computational demands of deep learning with the high-concurrency requirements of a Node.js backend. While limitations exist regarding hardware attestation and asynchronous queuing, the current iteration of AEGIS stands as a robust, production-ready, zero-trust distributed system.
